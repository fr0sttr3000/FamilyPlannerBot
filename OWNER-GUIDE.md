## 1. Управление пользователями

Доступ к боту контролируется через список `ALLOWED_USERS` в файле `.env`. Список содержит числовые Telegram ID через запятую.

### Как узнать Telegram ID пользователя

Попросите нового члена семьи:
1. Открыть Telegram
2. Найти бота `@userinfobot`
3. Отправить `/start`
4. Сообщить вам числовой ID из ответа (например: `987654321`)

### Добавить нового члена семьи

**Шаг 1.** Откройте файл `.env` на сервере:

```bash
cd /путь/к/FamilyPlannerBot
nano .env
```

**Шаг 2.** Найдите строку `ALLOWED_USERS` и добавьте новый ID через запятую:

```dotenv
# Было:
ALLOWED_USERS=123456789,987654321

# Стало (добавили 555000111):
ALLOWED_USERS=123456789,987654321,555000111
```

**Шаг 3.** Сохраните файл (`Ctrl+O`, `Enter`, `Ctrl+X` в nano).

**Шаг 4.** Перезапустите только бота (база данных не останавливается):

```bash
docker compose up -d --no-deps bot
```

**Шаг 5.** Проверьте что бот перезапустился:

```bash
docker compose ps
```

Новый пользователь сможет написать `/start` в течение 30 секунд.

> Всего время операции: 2–3 минуты.

### Удалить пользователя (заблокировать доступ)

**Шаг 1.** Откройте файл `.env`:

```bash
nano .env
```

**Шаг 2.** Удалите ID пользователя из строки `ALLOWED_USERS`:

```dotenv
# Было:
ALLOWED_USERS=123456789,987654321,555000111

# Стало (удалили 555000111):
ALLOWED_USERS=123456789,987654321
```

**Шаг 3.** Сохраните файл и перезапустите бота:

```bash
docker compose up -d --no-deps bot
```

После перезапуска пользователь немедленно теряет доступ. Его данные в базе сохраняются.

## 2. Ручной бэкап данных

Рекомендуется делать бэкап перед обновлением бота и перед переездом на новый хост.

```bash
cd /путь/к/FamilyPlannerBot
./scripts/backup.sh
```

Бэкап сохраняется в `./backups/` с именем вида `2026-05-09_03-00.sql.gz`.

Проверить созданные бэкапы:

```bash
ls -lh backups/
```

Бэкапы старше 30 дней удаляются автоматически при каждом запуске скрипта.

> Если настроен автоматический бэкап по расписанию (cron), ручной запуск всё равно работает и не мешает расписанию.

## 3. Восстановление данных

**Внимание:** Восстановление заменяет все текущие данные данными из бэкапа. Сделайте свежий бэкап перед восстановлением, если хотите сохранить текущее состояние.

**Шаг 1.** Посмотрите доступные бэкапы:

```bash
ls -lh backups/
```

**Шаг 2.** Запустите скрипт восстановления с путём к нужному файлу:

```bash
cd /путь/к/FamilyPlannerBot
./scripts/restore.sh ./backups/2026-05-09_03-00.sql.gz
```

Скрипт попросит подтверждение. Введите `yes` для продолжения.

Скрипт автоматически:
- Остановит бота
- Восстановит данные из бэкапа
- Запустит бота обратно

**Шаг 3.** Проверьте что бот работает:

```bash
docker compose ps
```

Напишите `/start` в Telegram — бот должен ответить.

## 4. Обновление бота

Обновление занимает 20–60 секунд. Данные не теряются.

```bash
cd /путь/к/FamilyPlannerBot

# 1. Сделать бэкап перед обновлением (рекомендуется)
./scripts/backup.sh

# 2. Получить новую версию кода
git pull

# 3. Обновить версию в .env (если нужно)
nano .env
# Изменить: APP_VERSION=v1.1.0

# 4. Пересобрать образ и перезапустить
docker compose build bot
docker compose up -d bot

# 5. Проверить
docker compose ps
docker compose logs --tail=20 bot
```

Если после обновления что-то сломалось — восстановите предыдущую версию из бэкапа (см. раздел 3).

## 5. Мониторинг

### Проверить что бот работает

```bash
docker compose ps
```

Ожидаемый вывод:
```
NAME      STATUS
fpb_bot   Up 2 hours
fpb_db    Up 2 hours (healthy)
```

Оба сервиса должны быть `Up`.

### Посмотреть логи бота в реальном времени

```bash
docker compose logs -f bot
```

Для выхода: `Ctrl+C`.

### Посмотреть последние 50 строк логов

```bash
docker compose logs --tail=50 bot
```

### Посмотреть логи за последний час

```bash
docker compose logs --since 1h bot
```

### Мониторинг потребления ресурсов

```bash
docker stats fpb_bot fpb_db
```

Ожидаемые значения:
- Бот: CPU < 5%, RAM < 100 МБ
- База данных: CPU < 2%, RAM < 200 МБ

Для выхода: `Ctrl+C`.

### Проверка занятого места на диске

```bash
# Место под данные PostgreSQL
docker system df

# Размер файлов бэкапов
du -sh backups/
```

## 6. Частые проблемы и решения

### Проблема 1: Бот не отвечает на команды

**Симптомы:** Сообщения отправляются, но бот молчит.

**Проверка:**
```bash
docker compose ps
docker compose logs --tail=30 bot
```

**Возможные причины и решения:**

| Причина | Решение |
|---|---|
| Контейнер бота упал | `docker compose up -d bot` |
| Ошибка подключения к Telegram API | Проверьте интернет: `curl -s https://api.telegram.org` |
| Неверный BOT_TOKEN | Проверьте `.env`, токен должен быть от @BotFather |
| Ваш ID не в ALLOWED_USERS | Добавьте ID согласно разделу 1 |

```bash
# Быстрое решение — перезапуск бота:
docker compose restart bot
```

### Проблема 2: Пользователь добавлен в ALLOWED_USERS, но бот его игнорирует

**Причина:** Изменения в `.env` вступают в силу только после перезапуска бота.

**Решение:**
```bash
docker compose up -d --no-deps bot
```

Подождите 20–30 секунд и попробуйте снова.

### Проблема 3: Бот был недоступен и потерял уведомления

**Ситуация:** Бот упал, за это время пришло время напоминания.

**Поведение:** Напоминания с временем в прошлом будут отправлены при следующем запуске бота. APScheduler обнаруживает пропущенные задания.

**Решение:** Убедитесь что бот работает:
```bash
docker compose ps
docker compose logs -f bot
```

Если бот упал — перезапустите: `docker compose up -d bot`.

### Проблема 4: Заканчивается место на диске

**Симптомы:** Ошибки в логах вида `no space left on device`.

**Проверка:**
```bash
df -h
docker system df
```

**Решения по приоритету:**

```bash
# 1. Удалить старые бэкапы (оставить последние 5)
ls -t backups/*.sql.gz | tail -n +6 | xargs rm -f

# 2. Очистить неиспользуемые Docker-образы
docker image prune -f

# 3. Очистить логи Docker (если очень большие)
# Проверить размер логов:
sudo du -sh /var/lib/docker/containers/*/

# 4. Изменить лимиты логов в docker-compose.yml (уже настроено: 10m × 3 файла = 30 МБ максимум)
```

### Проблема 5: После перезагрузки сервера бот не запустился

**Причина:** Docker daemon должен автозапускаться при старте системы.

**Проверка:**
```bash
systemctl is-enabled docker
# Ожидаемый ответ: enabled
```

**Включение автозапуска Docker:**
```bash
sudo systemctl enable docker
sudo systemctl start docker
```

**Запуск бота:**
```bash
cd /путь/к/FamilyPlannerBot
docker compose up -d
```

Чтобы бот стартовал автоматически после перезагрузки (без ручного вмешательства), Docker должен быть настроен на автозапуск (`systemctl enable docker`). Параметр `restart: unless-stopped` в docker-compose.yml гарантирует автозапуск контейнеров при старте Docker.

### Проблема 6: Сообщения не сохраняются / исчезают данные

**Проверка:**
```bash
docker compose ps
# fpb_db должен быть (healthy)

docker compose logs --tail=20 db
```

**Если контейнер БД упал:**
```bash
docker compose up -d db
# Подождать 20 секунд
docker compose restart bot
```

**Если volume с данными повреждён** (крайний случай):
1. Сделайте бэкап если возможно
2. Восстановите из последнего бэкапа: `./scripts/restore.sh ./backups/ПОСЛЕДНИЙ_ФАЙЛ.sql.gz`

### Проблема 7: Бот отвечает ошибкой на команды

**Симптомы:** Бот отвечает, но сообщает об ошибке.

```bash
# Проверить детали ошибки в логах
docker compose logs --since 5m bot | grep -i "error\|exception"
```

Если ошибка повторяется — обратитесь к разработчику с фрагментом лога.
